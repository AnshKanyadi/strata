# 0001. Push-based pipeline execution (vs pull-based Volcano)

## Status

Accepted

## Context

Strata is a vectorized, columnar analytical query engine. The previous ADRs establish that we process data in **cache-resident batches** (`DataChunk`s of `VECTOR_SIZE = 2048` values per column) rather than tuple-at-a-time. This ADR addresses a *separate, orthogonal* question: given that data flows through a tree of physical operators, **which operator drives the iteration — the consumer or the producer?**

Two canonical models exist:

- **Pull (Volcano / iterator model — Graefe).** Every operator exposes `next()`. The root calls `next()` on its child, which calls `next()` on *its* child, and so on down to the scan. Control flow is **demand-driven**: data moves up the tree because a parent asked for it. Each operator is a coroutine-like state machine that must suspend and resume across `next()` calls (e.g. a hash-join probe must remember where it was in the current build-side bucket chain). Note this resume-state bookkeeping is *not* a pull-only tax: push operators that emit more output than fits in one chunk per input must likewise carry position state and be re-invoked (this is precisely why DuckDB's `Execute` can return a `HAVE_MORE_OUTPUT` status and be re-called). The real difference is the *direction* of control, not the existence of operator state.

- **Push (produce/consume — Neumann, *Efficiently Compiling Efficient Query Plans for Modern Hardware*, PVLDB 2011).** The *source* drives. A scan reads a batch and it is pushed into a downstream `Consume(chunk)` method on the next operator, which transforms the batch in place (or emits a derived batch) and pushes it onward, until the batch reaches a **sink**: a hash aggregate, a join build side, a sort buffer, or the final result collector. Control flow is **data-driven**: data moves *forward* because the source produced it. Neumann frames this as compile-time `produce()` / `consume()` methods that *generate* one fused machine-code loop; in his formulation `consume()` conceptually handles a *tuple* inside the generated code — there are no runtime virtual batch calls. Strata adapts the same control-flow *direction* at runtime with `consume()` receiving a `DataChunk` (batch); that batch granularity is our vectorized adaptation, not Neumann's compiled formulation.

Concretely, the inversion is: in pull, the consumer holds the loop (`while (auto chunk = child->Next()) { ... }`); in push, the **executor/scheduler** holds the loop, repeatedly asking the source for a batch and pushing it through the operator chain — `while (auto chunk = source->GetData()) { sink_chain->Consume(chunk); }`. The loop owner in push is the executor, not the source operator running a self-contained loop; in the parallel case each worker runs this loop over morsels handed to it by the scheduler. Same dataflow graph, opposite call direction.

A **pipeline** in the push model is a maximal chain of operators with no intervening materialization: it runs from one source to one sink, streaming batches through fused, non-blocking operators (filters, projections, probe-side joins). A **pipeline breaker** is an operator that must consume its *entire* input before producing output — a hash-join build, a full aggregation, a sort. These breakers are exactly the sinks, and they delimit pipeline boundaries.

To be explicit, because it is the most common confusion at a whiteboard: **vectorization (batched execution) is orthogonal to push-vs-pull.** A Volcano iterator can return a 2048-row `DataChunk` from `next()` instead of one tuple — this is precisely what "vectorized Volcano" / MonetDB-X100-style engines do (Boncz, Zukowski, Nes, *MonetDB/X100: Hyper-Pipelining Query Execution*, CIDR 2005). A push engine can equally push one tuple at a time (that is Neumann's compiled model). This ADR decides **control-flow direction only**; batch granularity is settled elsewhere. The two choices compose freely, giving four quadrants — and real high-performance engines occupy more than one of them (see Photon below).

## Decision

**Strata uses a push-based execution model.** Sources push morsels of `DataChunk`s through a chain of operators into a sink. Operators implement a consume-style interface (`OperatorResult Execute(DataChunk& input, DataChunk& output)` for streaming operators; `void Sink(DataChunk& input)` for pipeline breakers). The query optimizer's physical plan is decomposed into pipelines at sink boundaries, and an executor drives each pipeline by repeatedly pulling batches from its single source and pushing them through.

### The decisive reason: morsel-driven parallelism

The marquee deliverable of this project (P8) is **morsel-driven parallelism** (Leis et al., *Morsel-Driven Parallelism*, SIGMOD 2014). Its model: input is sliced into small fixed-size **morsels**; a pool of worker threads (sized to the available cores) each grab a morsel via a **NUMA-aware, work-stealing-style dispatcher**, run it all the way through a pipeline, and report into shared sink state (e.g. a concurrent hash table). The paper's headline contribution is precisely this **NUMA-local morsel dispatch and scheduling**, not merely thread pinning. Parallelism is **per-pipeline and dispatch-driven**, not baked into the operator tree.

*Honest hardware caveat:* the dev machine (Apple M3 Pro — unified memory, a single NUMA domain, asymmetric 5 P-cores + 6 E-cores) means the **NUMA dimension is largely moot locally**, and "one identical worker per core" would ignore the P/E asymmetry. The morsel benefits we actually exercise on-device are **dynamic load balancing and cache locality**; the genuine NUMA story is validated on the multi-socket-style behavior of the x86_64 CI runner, where the dispatch-locality argument can actually bite.

Push is *natural* for this model and pull is *painful*:

- In **push**, a pipeline is already "a function from a source batch to side effects on a sink." A scheduler hands a worker a morsel; the worker pushes it through the fused operator chain; the only synchronization is at the sink. Multiple workers run the *same* pipeline over *different* morsels with no coordination between them until they touch shared sink state. The control flow already matches the parallelism model.
- In **pull**, the driving loop lives at the *top* of the tree (the root keeps calling `next()`). To parallelize, you must invert control anyway — either inject exchange operators (the Volcano-parallel approach, which fragments the plan, adds buffering/serialization, and couples the degree of parallelism into the plan shape), or retrofit a scheduler that re-drives stateful iterators from worker threads. Every operator's suspend/resume state must then become thread-safe or thread-local, which is exactly the friction the morsel model was designed to remove.

Neumann's produce/consume (PVLDB 2011) is the canonical push formulation, and is also the basis for data-centric **operator fusion** in *compiled* engines. **DuckDB**, the engine we benchmark against, itself moved from a pull (`GetChunk`) model to a **push-based** model in 2021 (see duckdb#1583), citing the difficulty of parallelizing multi-source pipelines (e.g. `UNION`), control-flow code duplicated across operators, and the inability to schedule operators independently of the tree — precisely the parallelism and pipelining pressures above. Matching DuckDB's control-flow direction lets us reason about the *same class* of design decisions DuckDB faces (pipeline scheduling, sink design) rather than attributing differences to a control-flow paradigm mismatch. This is a **pedagogical** alignment, not a fairness guarantee: it does **not** by itself make the benchmark apples-to-apples — storage, compression, optimizer, and hash-table maturity still differ — and we report those caveats with every number.

### Secondary wins

- **Tight per-batch loops.** A streaming operator receives a full `DataChunk` and runs a vectorized kernel over it before pushing. The hot loop is local to one operator invocation; there is no per-tuple virtual `next()` dispatch crossing operator boundaries.
- **In-pipeline intermediate avoidance (the vectorized version, stated honestly).** Within a pipeline we avoid materializing intermediates *to memory* between operators and use **selection vectors** to skip copies on filtered/joined rows. But — unlike Neumann-style *compiled* fusion, which collapses operators into one register-resident loop with no intermediate vectors — each **vectorized** operator still runs its own kernel and produces an output column or selection vector that the next operator reads. That *is* a form of intermediate (in-cache) materialization. This is exactly the compiled-vs-vectorized tradeoff dissected by Kersten et al., *Everything You Always Wanted to Know About Compiled and Vectorized Queries But Were Afraid to Ask* (PVLDB 2018): compiled = fewer instructions / one loop; vectorized = separate per-operator primitives with extra intermediate vectors but better SIMD utilization and latency hiding. We take the **vectorized side** and accept its extra per-operator intermediate vectors in exchange for simpler, SIMD-friendly kernels and an inspectable cost model.
- **Sinks as natural boundaries.** Pipeline breakers (build, aggregate, sort) are the sinks; the pipeline structure falls directly out of "where does a batch stop streaming and get materialized?" This makes pipeline lifetimes, buffer ownership, and "when is this intermediate state complete" easy to reason about.
- **Simpler pipeline lifetimes / scheduling unit.** A pipeline is a self-contained schedulable unit with explicit dependencies (a probe pipeline depends on its build pipeline finishing). This dependency DAG of pipelines is the scheduler's input — clean to express in push, awkward to extract from a recursive pull tree.

A pipeline is **complete** when all source morsels have been consumed, all workers have drained their in-flight batches into the sink, **and the sink's parallel state has been combined/finalized** (e.g. thread-local partial aggregates merged, partition builds completed, partial hash tables unioned). In the parallel/morsel setting, **sink finalization — not just source exhaustion — is the synchronization point**, and it is the genuinely hard part the morsel model introduces.

## Alternatives Considered

### Pull-based Volcano / iterator model
**Rejected.** The classic, well-understood model; trivially supports demand-driven early termination (just stop calling `next()`). But it makes the project's central deliverable — morsel-driven parallelism — a fight against the grain. Parallelizing pull requires either exchange operators (plan fragmentation, extra buffering, parallelism degree frozen into the plan) or making every operator's resume-state thread-safe. It also diverges from DuckDB's current architecture, reducing the pedagogical overlap of our benchmark study. The per-`next()` virtual dispatch and state-machine bookkeeping are additional (smaller) costs.

### Vectorized pull (Volcano returning batches, MonetDB-X100 / Photon style)
**Rejected as the control-flow choice — but explicitly *not* a strawman.** Returning a `DataChunk` from `next()` recovers most of the *vectorization* benefit and is a perfectly good engine design (it is essentially X100). Critically, it is also a **live commercial choice**: **Photon** (Behm et al., *Photon: A Fast Query Engine for Lakehouse Systems*, SIGMOD 2022) is a native, interpreted-vectorized engine whose operators use a `HasNext()`/`GetNext()` API to **pull** batches from their children — i.e. Photon sits in the *vectorized-pull* quadrant, the opposite of our choice. This is direct evidence that (a) the axes are genuinely orthogonal and (b) **pull is not unworkable** at the highest performance tier. Vectorized pull does **not**, however, change the *control-flow* problem this ADR decides: the loop still lives at the root, so the parallelism-ergonomics story is unchanged from plain pull. Since vectorization is orthogonal and already adopted, this option buys us nothing over plain pull on the only axis this ADR decides — and we chose push primarily for the morsel-scheduling ergonomics and DuckDB alignment, **not** because pull is infeasible.

### Push with per-tuple granularity (Neumann-style compiled, tuple-at-a-time push)
**Rejected.** This is the pure produce/consume model where the generated `consume()` handles one tuple, ideal when the whole pipeline is JIT-compiled into one register-resident loop so there is no per-tuple dispatch overhead. Strata is **interpreted-vectorized**, not compiled; pushing one tuple at a time would reintroduce exactly the per-tuple interpretation overhead that batching exists to amortize. We take push *control flow* with *batch* granularity — the DuckDB point in the design space.

### Async / coroutine-driven dataflow (C++20 coroutines as operators)
**Rejected for now.** C++20 coroutines could express operators as suspendable generators, blurring push/pull. But coroutine frame allocation and suspend/resume overhead are poorly suited to a 2048-wide hot loop, the abstraction obscures the cost model we want to teach, and it does not map as cleanly onto morsel scheduling as a plain push pipeline. Revisitable if we ever add async I/O sources; not justified for in-memory TPC-H.

## Consequences

### Wins (accepted benefits)
- Morsel-driven parallelism (P8) maps directly onto push pipelines; the scheduler's unit of work is "a morsel through a pipeline," with synchronization confined to sink state and its finalization.
- Control-flow alignment with DuckDB lets us study the same class of scheduling/sink decisions (a pedagogical win — explicitly not a benchmark-fairness guarantee).
- Tight, dispatch-free per-batch kernels and in-pipeline intermediate avoidance (selection vectors), with the vectorized tradeoff stated honestly.
- Pipelines and their dependency DAG are explicit, making lifetimes and scheduling tractable.

### Tradeoffs and costs (accepted honestly)
- **Inverted, less intuitive control flow.** "Where is the loop?" is no longer "at the root" — it is in the executor/scheduler. Reading a query's execution means thinking forward from sources, not backward from the root. New contributors (and the author at a whiteboard) must internalize this.
- **Early termination needs an explicit stop signal.** In pull, `LIMIT`/Top-N simply stops calling `next()` and the tree quiesces for free. In push, the executor is driving and the source does not know the consumer has had enough, so we must thread an explicit `OperatorResult::FINISHED` / "enough, stop pushing" signal back to the executor (which must check it between morsels). This is real plumbing we are signing up for, and it is the single sharpest ergonomic regression versus pull.
- **Some operators are arguably more natural in pull.** A **merge join** wants to advance whichever side is behind — a demand-driven "give me the next key from the smaller side" pattern that pull expresses directly; in push it becomes a small buffered state machine on each input. Certain **correlated / nested-loop** patterns (re-driving an inner pipeline per outer tuple) are likewise cleaner to *think about* as pull. We accept implementing these as buffered sinks / explicitly managed sub-pipelines in the push framework.
- **Backpressure is our responsibility.** Because the producer side drives, a fast source feeding a slow sink can over-produce in-flight batches; the model must bound this. (In pull, demand naturally throttles production.) For in-memory batch execution this is mild, but it is a property we now own rather than get for free.

## How to defend this at a whiteboard

- "Push vs pull is about **who holds the loop** — in push it's the executor/scheduler driving forward from the source; in pull it's the consumer pulling from the root. It is **orthogonal to vectorization**: you can have vectorized pull (that's MonetDB/X100, and that's **Photon** — `HasNext`/`GetNext`, interpreted vectorization) or tuple-at-a-time push (that's compiled Neumann). I chose push *control flow* with *batch* granularity, the DuckDB point."
- "The decisive reason is **morsel-driven parallelism** (Leis et al., SIGMOD 2014). A push pipeline is already 'a morsel in, side effects on a sink out,' so a scheduler just hands morsels to workers with synchronization only at the sink — and the pipeline is done only when source morsels are exhausted *and* the sink's parallel state is finalized. Retrofitting that onto stateful pull iterators means exchange operators or thread-safe resume-state — fighting the model."
- "**Neumann's produce/consume** (PVLDB 2011) is the canonical push formulation — but it's *compile-time* code-gen into one fused loop. I'm interpreted-vectorized, so I adapt the *direction* at runtime with `consume()` taking a `DataChunk`. Within a pipeline I avoid memory materialization and use selection vectors, but each vectorized operator still emits intermediate vectors — that's the compiled-vs-vectorized tradeoff from **Kersten et al., PVLDB 2018**, and I'm on the vectorized side."
- "**DuckDB moved from pull (`GetChunk`) to push in 2021 (duckdb#1583)** for multi-source pipeline scheduling and to stop duplicating control-flow code across operators. Matching that direction is a *pedagogical* alignment for studying the same decisions — it does **not** make the benchmark apples-to-apples; storage/optimizer/hash-table maturity still differ, and I caveat every number."
- "Citations I will not fumble: X100 is **Boncz, Zukowski, Nes (CIDR 2005)** — Kersten founded MonetDB but is *not* on that paper. **Photon is pull**, not push — it's real-world evidence pull stays viable, which is why I justify push on ergonomics, not on pull being broken."
- "Honest cost: control flow is inverted and **early termination needs an explicit stop signal** instead of just not calling `next()`. **Merge joins** and some correlated patterns are more natural in pull; I implement them as buffered state in the push framework. I also own backpressure. And operator resume-state isn't a pull-only tax — push operators that emit multiple chunks per input need a `HAVE_MORE_OUTPUT` re-invocation path too. I accept all of that for the parallelism win."
